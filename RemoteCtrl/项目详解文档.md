## 1. 项目概览

RemoteCtrl 是一个 **Windows 远程控制系统**，采用经典的 Client-Server 架构：

- **服务端 (RemoteCtrl)**：运行在被控机器上，监听 TCP 端口 9527，接收并执行来自客户端的命令
- **客户端 (RemoteClient)**：运行在控制者机器上，提供 MFC 图形界面，向服务端发送各种操控命令

技术栈：
- 语言：C++ (MFC 框架)
- 网络：Winsock 1.1，TCP 协议
- 图像：GDI+ / CImage (PNG 编码)
- 构建：Visual Studio 2022，x64 平台

## 4. 网络协议层

### 4.1 数据包格式 (CPacket)

**文件位置**：`RemoteCtrl/Packet.h`（服务端）/ `RemoteClient/CClientSocket.h`（客户端内嵌）

采用自定义二进制协议，小端序 (Little-endian)，使用 `#pragma pack(1)` 确保字节对齐：

```
┌──────────┬──────────┬──────────┬─────────────┬──────────┐
│  Header  │  Length  │ Command  │    Data     │ Checksum │
│  2 Bytes │  4 Bytes │  2 Bytes │  N Bytes    │  2 Bytes │
│  0xFEFF  │ N+4      │  CmdID   │  Payload    │  Sum     │
└──────────┴──────────┴──────────┴─────────────┴──────────┘
```

| 字段 | 大小 | 说明 |
|------|------|------|
| `sHead` | 2 bytes | 魔数 `0xFEFF`，用于定位包起始位置 |
| `nLength` | 4 bytes | `sizeof(sCmd) + sizeof(strData) + sizeof(sSum)` = N + 4 |
| `sCmd` | 2 bytes | 命令 ID (1-9, 1981) |
| `strData` | N bytes | 可变长度数据负载 |
| `sSum` | 2 bytes | 校验和 = 所有 Data 字节的累加和 (WORD) |

**总包大小**：`nLength + 6` 字节（Header 2 + Length 4 + 后续 nLength 字节）

**CPacket 类核心实现**：

```cpp
class CPacket {
public:
    WORD sHead;          // 固定 0xFEFF
    DWORD nLength;       // 数据长度
    WORD sCmd;           // 命令编号
    std::string strData; // 数据体
    WORD sSum;           // 校验和
    std::string strOut;  // 序列化输出缓冲

    // 构造方式 1：用于发送 —— 从命令和数据构建包
    CPacket(WORD nCmd, const BYTE* pData, size_t nSize);

    // 构造方式 2：用于接收 —— 从原始字节流解析包
    // nSize 作为引用参数：输入为缓冲区大小，输出为已消费的字节数
    // 如果数据不完整则设置 nSize = 0 表示解析失败
    CPacket(const BYTE* pData, size_t& nSize);

    int Size();           // 返回序列化后的总字节数
    const char* Data();   // 序列化为二进制数据
};
```

**解析流程** (接收端构造函数)：

```
1. 扫描字节流，寻找魔数 0xFEFF
2. 检查剩余数据是否足够（至少 i + 4 + 2 + 2 字节）
   → 不够：nSize = 0，返回（等待更多数据）
3. 读取 nLength，检查 nLength + i 是否超过 nSize
   → 超过：nSize = 0，返回（包未接收完整）
4. 读取 sCmd、strData、sSum
5. 计算校验和并比对
   → 匹配：nSize = 已消费字节数
   → 不匹配：nSize = 0
```

这个设计巧妙地通过 `nSize` 引用参数实现了**粘包处理**：调用者可以根据返回的 nSize 值，用 `memmove` 将已消费的数据移出缓冲区，剩余数据继续等待下一次解析。

## 5. 服务端 (RemoteCtrl)

### 5.1 程序入口 (RemoteCtrl.cpp)

**文件**：`RemoteCtrl/RemoteCtrl.cpp`（60 行）

服务端是一个**控制台 MFC 应用**（使用 `main()` 而非 `WinMain()`），启动流程：

```cpp
int main()
{
    // 1. 初始化 MFC 运行时
    AfxWinInit(hModule, nullptr, ::GetCommandLine(), 0);

    // 2. 创建命令处理器
    CCommand cmd;

    // 3. 启动服务器主循环（回调模式）
    int ret = CServerSocket::getInstance()->Run(&CCommand::RunCommand, &cmd);

    // 4. 错误处理
    switch (ret) {
    case -1: MessageBox(..."网络初始化异常"...); break;
    case -2: MessageBox(..."无法正常接入用户"...); break;
    }
}
```

#### 回调机制详解：网络层如何调用命令层？

这里的核心问题是：**CServerSocket（网络层）负责收数据，CCommand（命令层）负责处理数据，但它们是两个独立的类，CServerSocket 怎么把收到的命令"交给"CCommand 去执行？**

答案是**函数指针回调**。下面一步步拆解：

**第一步：定义回调函数的"形状"（函数指针类型）**

在 `ServerSocket.h` 的最顶部，定义了一个函数指针类型：

```cpp
// ServerSocket.h 第 9 行
typedef void (*SOCKET_CALLBACK)(void*, int, std::list<CPacket>&, CPacket&);
//              ↑                ↑     ↑    ↑                    ↑
//          函数指针类型名    通用指针  命令号  输出包列表          输入包
```

这行代码的含义：`SOCKET_CALLBACK` 是一个**函数指针类型**，它指向的函数必须长这样：
- 返回 `void`
- 接收 4 个参数：`void*`（通用指针）、`int`（命令号）、`list<CPacket>&`（输出）、`CPacket&`（输入）

**第二步：CServerSocket 保存回调指针，在 Run() 中调用它**

```cpp
class CServerSocket {
private:
    SOCKET_CALLBACK m_callback;  // 保存回调函数指针
    void* m_arg;                 // 保存传进来的通用指针

public:
    int Run(SOCKET_CALLBACK callback, void* arg, short port = 9527)
    {
        m_callback = callback;   // 存起来
        m_arg = arg;             // 存起来
        // ...
        while (true) {
            AcceptClient();
            int ret = DealCommand();        // 收到命令，ret = 命令号
            if (ret > 0) {
                // ★ 关键调用：通过函数指针调用外部传入的处理函数
                m_callback(m_arg, ret, lstPackets, m_packet);
                //         ↑      ↑       ↑          ↑
                //      &cmd对象  命令号  空列表(等填充) 收到的包

                // 发送 CCommand 填充好的响应包
                while (lstPackets.size() > 0) {
                    Send(lstPackets.front());
                    lstPackets.pop_front();
                }
            }
            CloseClient();
        }
    }
};
```

注意：**CServerSocket 完全不知道 CCommand 的存在**。它只知道"我有一个函数指针 `m_callback`，收到命令就调用它"。

**第三步：CCommand 提供一个符合签名的静态函数**

```cpp
class CCommand {
public:
    // 这个静态函数的签名 完全匹配 SOCKET_CALLBACK 的定义
    static void RunCommand(void* arg, int status, std::list<CPacket>& lstPacket, CPacket& inPacket)
    {
        // void* 还原为 CCommand*
        CCommand* thiz = (CCommand*)arg;

        // 调用成员函数处理命令
        thiz->ExcuteCommand(status, lstPacket, inPacket);
    }
};
```

**为什么需要 `void* arg` 和 `static`？**

这里有一个 C++ 的限制：**普通成员函数不能作为函数指针传递**，因为成员函数隐含一个 `this` 指针参数。所以必须用 `static` 函数（没有 `this`）。但 `static` 函数又无法访问成员变量，怎么办？

解决方案：通过 `void* arg` 把对象指针"偷渡"进去，在函数内部再强制转换回来：

```cpp
// arg 实际上就是 &cmd，转回 CCommand* 后就能调用成员函数了
CCommand* thiz = (CCommand*)arg;
thiz->ExcuteCommand(status, lstPacket, inPacket);
```

**第四步：main() 中把两者连接起来**

```cpp
int main()
{
    CCommand cmd;  // 创建命令处理器对象

    CServerSocket::getInstance()->Run(
        &CCommand::RunCommand,   // 参数1：函数指针（指向 CCommand 的静态函数）
        &cmd                     // 参数2：对象指针（通过 void* 传递）
    );
}
```

**完整调用链路图**：

```
main()
  │
  ├─ CCommand cmd;                          // 创建命令对象
  │
  └─ CServerSocket::Run(&RunCommand, &cmd)  // 传入函数指针和对象指针
       │
       │  m_callback = &CCommand::RunCommand  // 保存函数指针
       │  m_arg = &cmd                         // 保存对象指针
       │
       └─ while(true) 循环
            │
            ├─ AcceptClient()                  // 等待客户端连接
            ├─ DealCommand()                   // 收到命令包，返回命令号（如 6）
            │
            ├─ m_callback(m_arg, 6, lstPackets, m_packet)
            │   │
            │   │  等价于调用：
            │   │  CCommand::RunCommand(&cmd, 6, lstPackets, m_packet)
            │   │
            │   └─ 函数内部：
            │       CCommand* thiz = (CCommand*)arg;   // arg 还原为 &cmd
            │       thiz->ExcuteCommand(6, lstPackets, m_packet);
            │       │
            │       └─ m_mapFunction[6] → SendScreen()
            │           │
            │           └─ 截屏 → PNG 编码 → push 到 lstPackets
            │
            ├─ Send(lstPackets 中的所有响应包)  // 发送回客户端
            └─ CloseClient()
```

**为什么要这样设计？**

如果不用回调，最简单的做法是在 `CServerSocket::Run()` 里直接 `#include "Command.h"` 然后调用 `cmd.ExcuteCommand()`。但这样做的问题是：**网络层（CServerSocket）就依赖了命令层（CCommand）**，两者绑死在一起。

用回调函数后：
- `CServerSocket` 只依赖一个函数指针类型 `SOCKET_CALLBACK`，不需要知道 `CCommand` 的存在
- 如果将来想换一套命令处理器（比如 `CCommandV2`），只需要让新类也提供一个符合 `SOCKET_CALLBACK` 签名的静态函数，传给 `Run()` 即可，**CServerSocket 的代码一行都不用改**

这就是"解耦"的含义：两个模块通过一个约定好的接口（函数指针类型）交互，互相不知道对方的具体实现。

### 5.2 网络模块 (CServerSocket)

**文件**：`RemoteCtrl/ServerSocket.h`（224 行）/ `ServerSocket.cpp`（9 行，仅静态成员初始化）

**类职责**：TCP 服务器的完整生命周期管理 —— 初始化 Winsock、创建套接字、绑定端口、监听、接受连接、收发数据。

**单例模式实现**：

```cpp
class CServerSocket {
private:
    static CServerSocket* m_instance;  // 单例指针

    // 私有构造：初始化 Winsock 并创建 socket
    CServerSocket() {
        InitSockEnv();                          // WSAStartup(MAKEWORD(1,1))
        m_sock = socket(PF_INET, SOCK_STREAM, 0);
    }

    // 禁止拷贝和赋值
    CServerSocket(const CServerSocket&);
    CServerSocket& operator=(const CServerSocket&);

public:
    static CServerSocket* getInstance();

    // RAII 自动释放类
    class CHelper {
    public:
        CHelper()  { CServerSocket::getInstance(); }
        ~CHelper() { CServerSocket::releaseInstance(); }
    };
    static CHelper m_helper;  // 静态对象，程序退出时自动析构
};
```

`CHelper` 是一个**静态局部对象**，利用 C++ 静态对象的析构顺序，在程序退出时自动调用 `releaseInstance()` 释放单例，避免内存泄漏。

**核心方法 —— `Run()`**：

```cpp
int Run(SOCKET_CALLBACK callback, void* arg, short port = 9527)
{
    InitSocket(port);           // bind + listen
    while (true) {
        AcceptClient();          // 阻塞等待连接
        int ret = DealCommand(); // 接收并解析一个 CPacket
        if (ret > 0) {
            callback(arg, ret, lstPackets, m_packet);  // 调用命令处理回调
            // 发送所有响应包
            while (lstPackets.size() > 0) {
                Send(lstPackets.front());
                lstPackets.pop_front();
            }
        }
        CloseClient();          // 关闭连接，等待下一个
    }
}
```

**关键特点**：
- **单客户端模式**：每次只接受一个连接，处理完一个命令后关闭连接
- **阻塞式 I/O**：`accept()` 和 `recv()` 都是阻塞调用
- **4MB 接收缓冲区**：`#define BUFFER_SIZE 4096000`

**`DealCommand()` 实现**：

```cpp
int DealCommand()
{
    char* buffer = new char[BUFFER_SIZE];
    size_t index = 0;
    while (true) {
        size_t len = recv(m_client, buffer + index, BUFFER_SIZE - index, 0);
        if (len <= 0) return -1;
        index += len;
        len = index;
        m_packet = CPacket((BYTE*)buffer, len);  // 尝试解析
        if (len > 0) {                            // 解析成功
            memmove(buffer, buffer + len, BUFFER_SIZE - len);  // 移除已消费数据
            return m_packet.sCmd;                 // 返回命令 ID
        }
        // len == 0 表示数据不完整，继续 recv
    }
}
```

```cpp
    FILE* pFile = NULL;
    fopen_s(&pFile, strPath.c_str(), "rb");

    // 1. 获取文件大小
    fseek(pFile, 0, SEEK_END);
    long long data = _ftelli64(pFile);
    lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));  // 先发送 8 字节文件大小

    // 2. 分块发送文件内容
    fseek(pFile, 0, SEEK_SET);
    char buffer[1024] = "";
    size_t rlen = 0;
    do {
        rlen = fread(buffer, 1, 1024, pFile);
        lstPacket.push_back(CPacket(4, (BYTE*)buffer, rlen));
    } while (rlen >= 1024);   // 注意：>= 而非 >，修复了无限循环 bug

    fclose(pFile);
    return 0;
}
```

1. 第一个 Cmd 4 响应包：包含 8 字节的 `long long` 文件大小

```cpp
{
    MOUSEEV mouse;
    memcpy(&mouse, inPacket.strData.c_str(), sizeof(MOUSEEV));

    // 按键类型编码
    DWORD nFlags = 0;
    switch (mouse.nButton) {
    case 0: nFlags = 1;  break;  // 左键
    case 1: nFlags = 2;  break;  // 右键
    case 2: nFlags = 4;  break;  // 中键
    case 4: nFlags = 8;  break;  // 无按键（移动）
    }

    // 设置鼠标位置
    if (nFlags != 8) SetCursorPos(mouse.ptXY.x, mouse.ptXY.y);

    // 动作类型编码（与按键类型组合）
    switch (mouse.nAction) {
    case 0: nFlags |= 0x10; break;  // 单击
    case 1: nFlags |= 0x20; break;  // 双击
    case 2: nFlags |= 0x40; break;  // 按下
    case 3: nFlags |= 0x80; break;  // 释放
    }

    // 根据组合标志执行对应操作
    switch (nFlags) {
    case 0x21: /* 左键双击 */  → mouse_event(DOWN) + mouse_event(UP) × 2
    case 0x11: /* 左键单击 */  → mouse_event(DOWN) + mouse_event(UP)
    case 0x41: /* 左键按下 */  → mouse_event(LEFTDOWN)
    case 0x81: /* 左键释放 */  → mouse_event(LEFTUP)
    // 右键、中键类似...
    case 0x08: /* 仅移动 */    → mouse_event(MOUSEEVENTF_MOVE, x, y)
    }
}
```

**编码设计**：将按键类型（低 4 位）和动作类型（高 4 位）组合为一个字节，通过 switch-case 实现所有鼠标操作组合。

```cpp
{

    HDC hScreen = ::GetDC(NULL);                      // 获取整个屏幕的 DC
    int nBitPerpixel = GetDeviceCaps(hScreen, BITSPIXEL);
    int nWidth = GetDeviceCaps(hScreen, HORZRES);
    int nHeight = GetDeviceCaps(hScreen, VERTRES);

    screen.Create(nWidth, nHeight, nBitPerpixel);     // 创建与屏幕同大小的 CImage
    BitBlt(screen.GetDC(), 0, 0, nWidth, nHeight,
           hScreen, 0, 0, SRCCOPY);                   // 屏幕内容拷贝到 CImage
    ReleaseDC(NULL, hScreen);

    // 将 CImage 编码为 PNG 并写入内存流
    HGLOBAL hMem = GlobalAlloc(GMEM_MOVEABLE, 0);
    IStream* pStream = NULL;
    CreateStreamOnHGlobal(hMem, TRUE, &pStream);
    screen.Save(pStream, Gdiplus::ImageFormatPNG);     // 编码为 PNG

    // 从内存流中提取数据
    PBYTE pData = (PBYTE)GlobalLock(hMem);
    SIZE_T nSize = GlobalSize(hMem);
    lstPacket.push_back(CPacket(6, pData, nSize));     // 发送 PNG 数据

    // 清理资源
    GlobalUnlock(hMem);
    pStream->Release();
    GlobalFree(hMem);
    screen.ReleaseDC();
    return 0;
}
```

**与 CServerSocket 的主要区别**：

- 缓冲区使用 `std::vector<char>` 而非动态 `new char[]`
- `DealCommand()` 中使用 `static size_t index` 跨调用保持缓冲区状态

**单例模式**：与服务端相同的 CHelper + static instance 模式。

**连接初始化**：

```cpp
bool InitSocket(int nIP, int nPort)
{
    if (m_sock != INVALID_SOCKET) CloseSocket();  // 先关闭旧连接
    m_sock = socket(PF_INET, SOCK_STREAM, 0);

    sockaddr_in serv_adr;
    serv_adr.sin_family = AF_INET;
    serv_adr.sin_addr.s_addr = htonl(nIP);        // IP 地址大小端转换
    serv_adr.sin_port = htons(nPort);

    connect(m_sock, (sockaddr*)&serv_adr, sizeof(serv_adr));
    return true;
}
```

**数据接收 (`DealCommand()`)**：

```cpp
int DealCommand()
{
    char* buffer = m_buffer.data();
    static size_t index = 0;  // 跨调用持久化缓冲区索引

    while (TRUE) {
        // 优先消费缓冲区中已有的数据（避免连接关闭后丢失数据）
        if (index > 0) {
            size_t len = index;
            m_packet = CPacket((BYTE*)buffer, len);
            if (len > 0) {
                memmove(buffer, buffer + len, index - len);
                index -= len;
                return m_packet.sCmd;
            }
        }

        // 缓冲区中没有完整包，从网络接收更多数据
        int ilen = recv(m_sock, buffer + index, (int)(BUFFER_SIZE - index), 0);
        if (ilen == 0 || ilen == SOCKET_ERROR) return -1;
        index += (size_t)ilen;

        // 再次尝试解析
        size_t len = index;
        m_packet = CPacket((BYTE*)buffer, len);
        if (len > 0) {
            memmove(buffer, buffer + len, index - len);
            index -= len;
            return m_packet.sCmd;
        }
    }
}
```

**关键设计**：
- `static size_t index`：缓冲区索引是静态的，确保跨多次 `DealCommand()` 调用之间，残留在缓冲区中的未完整数据不会丢失
- **缓冲区优先消费**：先检查缓冲区中是否已有完整包（可能上次 recv 收了多个包），再 recv 新数据
- **`recv` 返回值处理**：使用 `int ilen` 接收 `recv` 返回值，避免 -1 转 `size_t` 导致的溢出问题

```cpp
        break;
    case 5:  // 鼠标事件（lParam 传 MOUSEEV 指针）
        ret = SendCommandPacket(cmd, autoClose, (BYTE*)lParam, sizeof(MOUSEEV));
        break;
    case 6: case 7: case 8:  // 屏幕截图 / 锁定 / 解锁
        ret = SendCommandPacket(cmd, autoClose);
        break;
    }
    return ret;
}
```

## 7. 线程模型

### 服务端线程

```
主线程 (Main Thread)
│
├─ while(true) 循环
│    ├─ AcceptClient()     ← 阻塞等待
│    ├─ DealCommand()      ← 接收命令
│    ├─ RunCommand()       ← 执行命令
│    ├─ Send(响应包)        ← 发送结果
│    └─ CloseClient()      ← 关闭连接
│
└─ 锁屏线程 (threadLockDlg)  ← Cmd 7 时 _beginthreadex 创建
     ├─ 创建全屏对话框
     ├─ 限制鼠标/隐藏光标/隐藏任务栏
     └─ GetMessage() 循环等待 'A' 键解锁
```

### 客户端线程

```
主线程 (UI Thread)
│
├─ MFC 消息循环
├─ UI 事件处理
├─ WM_SEND_PACKET 消息处理（子线程请求发包）
│
├─ 监控线程 (threadWatchData)  ← "开始监控" 按钮 _beginthread 创建
│    ├─ 每 50ms 发送 Cmd 6 请求屏幕截图
│    ├─ 接收 PNG 数据
│    ├─ 加载到 CImage 缓存
│    ├─ 设置 m_isFull = true
│    └─ m_isClosed == true 时退出
│
└─ 下载线程 (threadDownFile)   ← "下载文件" 菜单 _beginthread 创建
     ├─ 发送 Cmd 4 请求
     ├─ 接收文件大小
     ├─ 循环接收数据块并写入本地文件
     └─ 完成后关闭 socket
```

**线程同步机制**：
- **m_isFull 标志**：监控线程设置为 true（有数据），定时器消费后设置为 false（可接收新数据）
- **m_isClosed 标志**：主线程在对话框关闭后设置为 true，通知监控线程退出
- **WaitForSingleObject(hThread, 500)**：等待监控线程退出，超时 500ms
- **SendMessage**：子线程通过同步消息 `SendMessage` 回到主线程执行网络操作，避免跨线程 socket 竞争

## 9. 设计模式

| 模式 | 应用场景 | 具体实现 |
|------|---------|---------|
| **单例模式 (Singleton)** | CServerSocket, CClientSocket | 静态指针 + CHelper RAII 释放 |
| **命令模式 (Command)** | CCommand 命令分发 | `map<int, CMDFUNC>` 映射命令 ID 到函数指针 |
| **回调模式 (Callback)** | 网络层与命令层解耦 | `SOCKET_CALLBACK` 函数指针，Run() 中调用 |
| **观察者模式 (Observer)** | 屏幕截图显示 | 监控线程更新缓存，Timer 轮询显示 |
| **代理模式 (Proxy)** | 跨线程通信 | WM_SEND_PACKET 消息，子线程通过主线程代理发送网络命令 |

### 单例 + CHelper 模式详解

```cpp
class CServerSocket {
    static CServerSocket* m_instance;

    class CHelper {
        CHelper()  { CServerSocket::getInstance(); }  // 程序启动时创建
        ~CHelper() { CServerSocket::releaseInstance(); } // 程序退出时释放
    };
    static CHelper m_helper;
};
```

`m_helper` 是**静态成员**，在程序加载时构造，退出时析构。这利用了 C++ 的 RAII 特性，确保单例对象在程序结束时被正确释放（调用 `WSACleanup()` 和 `closesocket()`）。

### WM_SEND_PACKET 消息代理

为什么不让子线程直接调用 `SendCommandPacket()`？

因为 MFC 的 UI 操作必须在主线程执行，而 `SendCommandPacket()` 中调用了 `UpdateData()` 读取 UI 控件。通过 `SendMessage` 将操作转发到主线程，既保证了线程安全，又避免了对 socket 的并发访问。

```
子线程                     主线程
  │                          │
  ├─ SendMessage() ────────→ │ OnSendPacket()
  │  (同步等待)               │  ├─ SendCommandPacket()
  │                          │  │  ├─ UpdateData()
  │                          │  │  ├─ InitSocket()
  │                          │  │  ├─ Send()
  │                          │  │  └─ DealCommand()
  │  ←── 返回结果 ─────────── │  └─ return ret
  └─ 继续执行
```

## 11. 已知设计问题

| 问题 | 描述 | 影响 | 可能的改进方向 |
|------|------|------|--------------|
| **网络与 UI 紧耦合** | CWatchDialog 直接通过 `GetParent()` 调用 CRemoteClientDlg 方法 | 难以测试和复用 | 引入 MVC/观察者模式解耦 |
| **单客户端限制** | 服务端一次只能接受一个 TCP 连接 | 无法多用户控制 | 改用多线程/异步 I/O |
| **短连接模式** | 大部分命令都是连接→发送→接收→关闭 | 频繁建立/断开连接开销大 | 改为长连接 + 命令队列 |
| **无加密** | 所有数据明文 TCP 传输 | 安全风险 | 加入 TLS 或自定义加密层 |
| **无认证** | 任何客户端都可连接 9527 端口 | 安全风险 | 添加密码认证握手 |
| **4MB 缓冲区** | 固定大小缓冲区 | 大文件/高分辨率截图可能溢出 | 动态扩展缓冲区 |
| **协议代码重复** | CPacket 在服务端和客户端各一份 | 修改时容易不同步 | 提取为共享库/头文件 |
| **锁屏安全性** | 通过全屏窗口 + ClipCursor 实现 | Alt+Tab/Ctrl+Alt+Del 可能绕过 | 更底层的锁定方式 |
